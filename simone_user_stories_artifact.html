EPIC-01
Caller Identification & Intelligent Routing
Core Infrastructure
US-101
ANI Lookup & Patient Record Retrieval
Must Have
As a returning patient calling to manage an appointment,
I want the system to automatically recognize my phone number and retrieve my record,
So that I don't have to re-enter my identifying information and can be routed immediately to the correct service.
Acceptance Criteria
  • AC-01Given a caller dials in from a registered phone number, when the ANI lookup executes, then the system retrieves the matching patient record from the EHR/PMS within 2 seconds of call answer.
  • AC-02Given a patient record is successfully retrieved, when the IVA greets the caller, then the greeting includes the patient's preferred first name (e.g., "Hi Sarah, welcome back").
  • AC-03Given a successful match, when routing logic executes, then the caller is directed to their primary care provider's scheduling queue within one IVA turn — no additional identification steps required.
  • AC-04Given the ANI lookup completes, the system must log the lookup result (match/no-match/error), timestamp, and caller ID to the audit trail for compliance.
  • AC-05Given PMS data is returned, the IVA must display the patient's next upcoming appointment (if any) as a contextual prompt during the interaction.
Edge Cases
  • EC-01If the ANI returns multiple patient records (e.g., shared household phone), the IVA must prompt for date of birth verification before proceeding — maximum 2 verification attempts before transferring to a live agent.
  • EC-02If the ANI lookup times out after 3 seconds, the IVA must gracefully fall through to the new patient identification flow without exposing a system error to the caller.
  • EC-03If a patient has opted out of personalized greetings (HIPAA preference flag), the system must default to a generic greeting and suppress name and appointment data.
  • EC-04If the PMS returns a "deceased" patient status, the IVA must not proceed with scheduling — transfer to a live agent with a silent flag for sensitivity handling.
QA Validation Criteria
  • QA-01Execute ANI lookup with 50 known registered numbers — verify 100% match rate and correct name retrieval across all EHR record types (Epic, Athena, NextGen).
  • QA-02Simulate lookup latency at 1s, 2s, 2.9s, and 3.1s — confirm fallthrough triggers only at >3s and that greeting is not delivered before record is confirmed.
  • QA-03Test shared household phone scenario with 3 records — confirm DOB prompt appears, incorrect entry is handled gracefully, and 3rd failure routes to agent.
  • QA-04Verify audit log entries are written for 100% of lookup attempts — confirm HIPAA opt-out flag suppresses PII in greeting across 10 flagged test accounts.
Dependencies & Notes
EHR/PMS API Integration ANI Feed Provisioned HIPAA Flag Schema v2.1 Agent Transfer Queue Config Audit Log Service IVA Platform: Intelepeer
US-102
New Patient Identification & Verification Flow
Must Have
As a new patient or caller without a registered phone number,
I want the IVA to guide me through a brief, dignified verification sequence,
So that I can access the right services without repeating myself multiple times or being abandoned in a loop.
Acceptance Criteria
  • AC-01Given ANI lookup returns no match, when the IVA initiates the new-caller flow, then the caller is greeted neutrally (no name assumption) and prompted to confirm if they are a new or existing patient.
  • AC-02Given the caller confirms they are an existing patient, when the IVA requests verification, then it collects date of birth and last name only — no more than 2 data fields before routing.
  • AC-03Given the caller confirms they are a new patient, when the IVA responds, then it offers exactly two options: schedule a new patient appointment or transfer to the registration team — presented within a single IVA turn.
  • AC-04Given successful verification of an existing patient by DOB + last name, when the record is confirmed, then the IVA resumes the same personalized experience as a matched ANI caller (AC-02 through AC-05 in US-101 apply).
Edge Cases
  • EC-01If DOB + last name verification fails on first attempt, the IVA offers one retry with a different prompt phrasing. On second failure, the caller is transferred to a live agent — no third attempt.
  • EC-02If the caller does not respond within 5 seconds of a prompt, the IVA re-prompts once, then offers a callback option — the call must not terminate silently.
  • EC-03If verification returns multiple matching records (same DOB + last name), escalate to agent immediately with a silent priority flag — do not attempt further disambiguation via IVA.
QA Validation Criteria
  • QA-01Test all four entry states: new patient, existing patient verified, existing patient fail (1x), existing patient fail (2x) — confirm correct routing and message phrasing for each.
  • QA-02Validate that the verification flow never exceeds 2 data collection turns before routing — time each scenario and confirm average handle time is under 45 seconds to route.
  • QA-03Test silence/timeout path — confirm re-prompt fires at 5s and callback option appears at second silence event with correct callback confirmation message.
Dependencies & Notes
EHR Verification API Registration Team Queue Callback Service Config Silence Detection (5s threshold)
US-103
Dynamic Intent Classification & Call Routing
Must Have
As a patient who has been identified by the system,
I want to state the reason for my call in plain language and be routed immediately,
So that I reach the right team or complete my task without navigating a lengthy menu tree.
Acceptance Criteria
  • AC-01Given an identified patient states their intent, when the NLP intent classifier processes the utterance, then it correctly classifies intent into one of 6 defined categories (Schedule, Cancel, Reschedule, Confirm, Billing, Other) with ≥90% accuracy on the production test set.
  • AC-02Given a classified intent of Schedule, Cancel, or Reschedule, when routing executes, then the caller reaches the correct queue or self-service path in a single IVA step — no confirmation menus.
  • AC-03Given an intent of "Billing" or "Other," when routing executes, then the caller is transferred to the appropriate live queue within 8 seconds with a warm handoff message including the patient's name and intent.
  • AC-04Given the IVA cannot classify an intent with >70% confidence, when fallback logic triggers, then the caller is offered a short-form menu of the 3 most common intents — not a full menu replay.
Edge Cases
  • EC-01If a caller states multiple intents in one utterance (e.g., "cancel my appointment and also ask about my bill"), the IVA must handle the primary intent first and offer to address the secondary intent after completion.
  • EC-02If the caller uses a non-English utterance, the IVA must detect the language and route to a bilingual queue immediately — Spanish (Castilian) and Portuguese to be supported at MVP; other languages route to agent.
  • EC-03If a caller utters keywords associated with a medical emergency, the IVA must immediately play an emergency redirect message and transfer to 911 instructions — no scheduling flow is permitted to continue.
QA Validation Criteria
  • QA-01Run the NLP classifier against 200-utterance test set (balanced across all 6 intent categories) — confirm ≥90% classification accuracy; document and categorize all misclassifications for model tuning.
  • QA-02Test multi-intent utterances (minimum 15 scenarios) — verify primary intent is handled first and secondary intent offer appears post-completion in all cases.
  • QA-03Test emergency keyword list (minimum 8 trigger phrases) — confirm 100% emergency redirect rate with zero scheduling flow continuations after trigger detection.
  • QA-04Test Spanish and Portuguese utterances across all 6 intent categories — confirm bilingual queue routing fires correctly for each.
Dependencies & Notes
NLP Intent Model v3.2 Bilingual Queue Setup Emergency Keyword Library Warm Handoff API Confidence Threshold Config (0.70)
EPIC-02
Appointment Management Self-Service
Scheduling Core
US-201
Appointment Scheduling via Live PMS Availability
Must Have
As a patient who wants to schedule an appointment,
I want the IVA to query real-time availability from the PMS and confirm my appointment in one call,
So that I don't need to call back, wait on hold, or receive a callback to complete scheduling.
Acceptance Criteria
  • AC-01Given a patient requests scheduling, when the IVA queries the PMS, then it presents up to 3 available appointment slots in the patient's preferred provider's schedule — ordered by soonest available.
  • AC-02Given the patient selects a time slot, when confirmation executes, then the appointment is written to the PMS in real time and a confirmation number is read back to the caller and sent via SMS within 60 seconds.
  • AC-03Given the patient has a stated appointment type preference on file (in-person vs. telehealth), when slots are presented, then the IVA filters availability to match the preference and notes the appointment modality before confirming.
  • AC-04Given a successful booking, when the call ends, then the PMS record is updated with the IVA interaction source code for reporting and audit purposes.
  • AC-05Given the caller requests a provider not in network or not currently accepting patients, when the IVA responds, then it offers the next available in-network provider and does not dead-end the interaction.
Edge Cases
  • EC-01If no availability exists within 14 days for the patient's preferred provider, the IVA must offer a waitlist enrollment option and confirm enrollment by reading back contact details.
  • EC-02If PMS write fails after slot selection (timeout or API error), the IVA must not confirm the appointment — instead it must inform the caller, attempt one retry, and offer agent transfer on second failure.
  • EC-03If the patient has an existing appointment within 24 hours of the requested new slot, the IVA must surface the conflict and ask for confirmation before proceeding.
  • EC-04If the patient requires an interpreter (language flag in PMS), the IVA must include the interpreter service code in the booking and notify the scheduling team via internal alert.
QA Validation Criteria
  • QA-01Perform 25 end-to-end scheduling test calls — confirm PMS write completes successfully and appointment appears in chart within 60 seconds for all 25.
  • QA-02Test PMS write failure scenario (force timeout) — confirm IVA does not read confirmation number, retry fires once, and agent transfer triggers on second failure.
  • QA-03Test slot conflict detection with 10 accounts that have existing upcoming appointments — confirm conflict prompt appears in 100% of cases before booking proceeds.
  • QA-04Confirm SMS confirmation fires within 60 seconds for 100% of completed bookings across iOS and Android test numbers.
Dependencies & Notes
PMS Real-Time Write API SMS Gateway Config Waitlist Service Interpreter Flag Schema IVA Source Code: IVA-SCHED Conflict Detection Logic
US-202
Appointment Cancellation with Reason Capture
Must Have
As a patient who needs to cancel an upcoming appointment,
I want to cancel quickly via the IVA and have the option to reschedule in the same interaction,
So that I don't lose my care continuity and the provider's schedule is updated immediately.
Acceptance Criteria
  • AC-01Given a patient requests cancellation, when the IVA retrieves their upcoming appointments, then it presents the soonest appointment first with date, time, and provider name — and confirms the correct appointment before proceeding.
  • AC-02Given the patient confirms cancellation, when the IVA captures a cancellation reason (from a list of 5 structured options), then the PMS appointment status is updated to "Cancelled – IVA" and the reason code is appended within 30 seconds.
  • AC-03Given cancellation is confirmed, when the IVA response concludes, then the caller must be offered reschedule options before the call ends — not as a separate call.
  • AC-04Given a cancellation within 24 hours of the appointment, when the IVA confirms, then an internal alert must be sent to the provider's scheduling coordinator for same-day slot backfill opportunity.
Edge Cases
  • EC-01If the patient has no upcoming appointments in the PMS, the IVA must confirm this clearly and offer to connect with the scheduling team — not present an empty menu.
  • EC-02If the patient has multiple upcoming appointments and selects incorrectly, the IVA must offer one re-confirmation before executing — "Just to confirm, you'd like to cancel your [date/time] appointment with [Dr. X]?"
  • EC-03If the appointment type is flagged as "Pre-procedure" or "Post-operative," the IVA must play a clinician advisory message before accepting the cancellation and log the interaction for clinical staff review.
QA Validation Criteria
  • QA-01Test cancellation flow for accounts with 1, 2, and 3+ upcoming appointments — confirm correct appointment is cancelled in all cases and PMS status updates within 30 seconds.
  • QA-02Validate all 5 cancellation reason codes are correctly written to PMS across 5 distinct test calls — one per reason.
  • QA-03Test pre-procedure flag: confirm advisory message plays before cancellation is accepted and clinical alert fires for 100% of flagged appointment types.
  • QA-04Test same-day cancellation (within 24 hours) for 5 test appointments — confirm scheduling coordinator alert is received within 2 minutes of IVA confirmation.
Dependencies & Notes
PMS Cancel Status API Cancellation Reason Code Library (5) Internal Alert Service Pre-Procedure Flag Schema Clinical Escalation Queue
US-203
Appointment Confirmation & Reminder Re-Engagement
Should Have
As a patient who received an automated appointment reminder,
I want to confirm, cancel, or reschedule my appointment in response to that reminder via IVA,
So that my intent is captured in real time and my provider is notified without requiring a manual callback.
Acceptance Criteria
  • AC-01Given a caller dials in referencing a reminder (via utterance OR callback number match), when the IVA detects the reminder context, then it presents the specific appointment details and asks for confirmation, cancellation, or reschedule — in a single turn.
  • AC-02Given a caller confirms the appointment, when the IVA processes the response, then the PMS appointment status is updated to "Patient Confirmed – IVA" and confirmation is re-sent via SMS within 60 seconds.
  • AC-03Given a caller confirms, when the call ends, then the IVA offers one add-on prompt: "Would you like us to remind you again 2 hours before?" — and records the preference.
Edge Cases
  • EC-01If the appointment was already confirmed via another channel (patient portal, text), the IVA must acknowledge the prior confirmation and not create a duplicate status write to the PMS.
  • EC-02If the reminder was for an appointment that has since been cancelled by the practice, the IVA must surface this immediately, apologize for the confusion, and offer to reschedule.
QA Validation Criteria
  • QA-01Test inbound re-engagement from 10 reminder callbacks — confirm context detection fires correctly and appointment details presented match PMS for all 10.
  • QA-02Test duplicate confirmation prevention — confirm PMS does not write a second "confirmed" status when appointment already shows confirmed; log deduplication event.
Dependencies & Notes
Reminder System Callback Mapping PMS Confirmation Status Write Deduplication Logic 2-Hour Reminder Preference Store
EPIC-03
Latency Reduction & System Reliability
Performance
US-301
IVA Response Latency & Turn Optimization
Must Have
As a patient on a call with the IVA,
I want the system to respond to my inputs quickly and without noticeable lag,
So that the interaction feels natural rather than like I am waiting for a broken phone system.
Acceptance Criteria
  • AC-01Given a caller finishes speaking, when the IVA processes the utterance, then the audio response begins playing within 1.5 seconds of end-of-speech detection — measured at the 90th percentile across production calls.
  • AC-02Given any IVA turn requiring an API call (ANI lookup, PMS query, scheduling write), when the response is delayed beyond 2 seconds, then the IVA plays a brief filler message ("One moment…") — not silence.
  • AC-03Given a complete call flow from answer to task completion, when the call ends, then average handle time for scheduling tasks must not exceed 3 minutes 30 seconds as measured in production telemetry.
  • AC-04Given the IVA processes barge-in (caller speaks over the IVA), when barge-in is detected, then the IVA response stops immediately and the caller input is re-processed — zero blocking of barge-in.
Edge Cases
  • EC-01If API latency exceeds 5 seconds on any call, the system must log the event as a "high latency incident" and trigger an alert to the operations monitoring dashboard — no silent failures.
  • EC-02If a filler message plays more than twice in a single call, the IVA must route to a live agent rather than continue degraded — flagging the call for engineering review.
QA Validation Criteria
  • QA-01Conduct latency benchmarking across 100 test calls — confirm P90 response time ≤1.5s; document P95 and P99 for engineering baseline.
  • QA-02Test barge-in in 20 scenarios — confirm 100% barge-in detection and IVA cessation within 200ms of barge-in detection.
  • QA-03Force API latency >5s via test environment — confirm high-latency alert fires in monitoring dashboard within 30 seconds of event.
Dependencies & Notes
Telemetry / Call Monitoring Dashboard Barge-In Detection Config Filler Audio Assets Ops Alert Service Latency SLA: 1.5s P90
US-302
Graceful Failure & Agent Escalation Protocol
Must Have
As a patient experiencing a system error or unresolvable IVA interaction,
I want to be transferred to a live agent smoothly with context already handed off,
So that I don't have to re-explain my situation from the beginning.
Acceptance Criteria
  • AC-01Given any escalation trigger (patient request, max retries, system error, or clinical flag), when the transfer executes, then the receiving agent screen-pops with: patient name, verified identity status, intent classification, steps completed, and reason for escalation.
  • AC-02Given a transfer is initiated, when the caller is in queue, then the IVA plays a hold message (not silence or dead air) with an estimated wait time drawn from live queue data.
  • AC-03Given a caller requests to speak to a live agent at any point in the call flow, when the request is detected (via utterance or keypress), then escalation initiates immediately — no additional IVA turns or re-confirmation required.
  • AC-04Given the queue wait time exceeds 4 minutes, when the IVA detects the threshold, then it offers the caller a callback option — capturing the preferred callback number and confirming enrollment.
Edge Cases
  • EC-01If the agent queue is closed or at maximum capacity, the IVA must not place the caller in an endless hold — offer callback enrollment and end the call gracefully within one additional turn.
  • EC-02If the screen pop data fails to transmit (API error), the transfer still proceeds — a flag is written to the call log and the agent is notified of the missing context via a system message in the agent UI.
QA Validation Criteria
  • QA-01Test 5 distinct escalation triggers — confirm screen pop populates correctly for all 5 in agent UI with no missing required fields.
  • QA-02Test "talk to an agent" utterance at 5 different points in the IVA flow — confirm transfer initiates without any additional IVA turns in all 5 scenarios.
  • QA-03Simulate queue >4 minutes — confirm callback offer appears and enrollment completes with correct callback number confirmed back to caller.
  • QA-04Test screen pop failure — confirm transfer still completes and agent-side system message appears flagging missing context.
Dependencies & Notes
Agent Screen Pop Integration Live Queue Data Feed Callback Enrollment Service Agent UI Alert System Queue Threshold Config: 4 min
