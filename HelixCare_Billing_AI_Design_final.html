PHI-Safe Billing Flow
Problem Statement & Background

Current State

HelixCare's billing inquiries and payment transactions are handled entirely through live agent queues, generating high call volumes for routine balance and payment interactions. Patients experience wait times averaging 8–12 minutes for requests that require no clinical judgment. Billing staff spend an estimated 60% of inbound call time on balance reads and payment captures that could be fully automated.

Desired Outcome

Deploy a HIPAA-compliant AI assistant capable of handling balance inquiries, full and partial payments, payment plan presentations, and billing FAQs without live agent involvement. The system must maintain PHI guardrails throughout, integrate with HelixCore PMS for real-time data, and execute clean warm handoffs when automation is not appropriate.

Scope of This Document

This document covers the functional requirements, orchestration logic, conversation design, API integration specifications, security guardrails, KPI definitions, and open items required to build, test, and launch the HCAI billing module across voice, web chat, and SMS channels. It is the source of truth for product, engineering, QA, and operations alignment.

APIs for Truth
Balances, due dates, payment outcomes, and account attributes are never generated by the model. All transactional facts are retrieved from HelixCare APIs or billing systems of record.
LLM for Reasoning
The LLM classifies intent, handles clarifications, rewrites technical content into human language, and adapts to channel context. It does not invent or finalize financial outcomes.
Auth Before PHI
Any balance, due date, payment history, or billing arrangement requires identity verification. FAQ-only interactions can remain unauthenticated when no PHI is exposed.
Fail Gracefully
API errors, low-confidence intents, tokenization failures, and auth failures all have defined fallback paths. The system never leaves the user in a conversational cul-de-sac.
Design for Adoption
The assistant sounds helpful, direct, and respectful. It avoids stiff compliance theater while still handling disclosure, security, and escalation in a way users can tolerate.

Orchestration Layer — The System Brain
All billing and payment interactions are governed by a centralized orchestration layer. This layer decides when to invoke APIs, when to let the LLM respond, when authentication is required, when to escalate, and how to recover from failures. This is the control tower. The assistant's tone may feel conversational, but the operating model underneath is stubbornly deterministic where it matters.
Execution Priority
1. Security + policy checks
2. API retrieval for financial facts
3. Rules engine for routing
4. LLM for phrasing and clarification
Confidence Logic
≥ 0.85 confidence: proceed to mapped flow
0.60–0.84: ask one clarification
< 0.60: offer guided options or agent
Repeat confusion: immediate escalation
Recovery Logic
API failure: retry twice with backoff
Payment failure: alternate method or agent
Auth failure: generic message, no PHI disclosure
User frustration: bypass automation

LLM Role Boundary: The model may summarize, clarify, and soften system language, but it cannot fabricate balances, dates, payment confirmations, or policy answers when source data is unavailable. If truth is missing, the experience must say so plainly and route to the next safest option.

Acquire Tokens, Resolve Session Context, and Prepare Tool Access
Before the assistant acts on billing workflows, platform credentials and channel metadata must already be in place. Web chat sessions use authenticated user context when available. Voice sessions rely on ANI and session tokens. SMS sessions reuse the most recent verified context only within a defined expiry window. Token refresh is proactive. Secrets never touch logs.

Identity & Authentication · PHI Gate · Retry Rules
Verify Identity Before Showing Balance or Taking Payment
Any balance, invoice amount, due date, saved payment method, or payment plan eligibility requires successful identity verification. If ANI or signed context suggests a likely user, verification can be shorter. If not, the assistant gathers identifiers and verifies them through HelixCare APIs. Maximum of two failed attempts before transfer. Failure language remains generic to avoid confirming whether a record exists.

Balance Retrieval · API Truth · Decisioning
Retrieve the Balance and Decide the Next Best Action
Once identity is confirmed, the system retrieves current balance, due date, eligible invoices, and payment-plan flags. The assistant then adapts the next step based on returned data. A zero balance does not trigger payment prompts. A delinquent balance can trigger a payment-plan offer. API responses drive the flow, not generic copy.
If Balance = 0
Inform user no balance is due
Offer email or SMS receipt if recent payment exists
End or route to billing FAQ
If Balance > 0
Read total due and due date
Offer full or partial payment
Offer plan if eligible
If API Fails
Retry twice with backoff
Fallback message without making up numbers
Warm transfer to billing
Canonical Balance Script
"Your current balance is $142.50, and it is due on March 30. Would you like to pay this amount, make a different payment, or hear your payment plan options?"
🧠
Decision-Aware API Use: The system does not just fetch a balance. It interprets the response. If balance is zero, it avoids a dead-end "Would you like to pay?" moment. If planEligible is true, it introduces a plan option at exactly the point it becomes relevant.

Payment Decision · Amount Selection · Fee Disclosure
Let the User Choose Full, Partial, or Plan-Based Payment
After the balance is read, the assistant offers the clearest next moves: pay in full, pay a different amount, or hear plan options. If a fee applies for card or channel usage, the system discloses that before capture. Minimum payment thresholds may exist in the backend, but the assistant should only surface what policy allows users to act on.

Payment Capture · Tokenization · Charge Execution
Capture Payment Securely and Resolve the Transaction
The assistant collects payment through a PCI-safe capture experience. Card data is tokenized immediately and never retained in conversational logs. Once tokenization succeeds, the charge request is submitted. If tokenization or charging fails, the assistant offers a retry, alternate method, or billing handoff.
Success Path
Tokenize payment method
POST charge request
Receive transaction ID
Proceed to confirmation
Failure Path
Tokenization error: restart capture once
Charge failure: offer alternate amount or method
Repeated failure: transfer to billing
Log reason code internally
!
Security Guardrail: Raw payment data must never be displayed back to the user or logged in conversation transcripts. The assistant may confirm last-four digits only if returned by the tokenization service in a compliant manner.

Fallback and Human Handoff
When the system cannot verify identity, cannot retrieve trustworthy data, or cannot complete payment safely, the assistant exits automation cleanly and carries the context into a human handoff.
7
Escalation · Warm Transfer · Context Preservation
Hand Off Without Making the User Repeat Themselves
Escalation is not a failure of design. It is part of the design. When the assistant transfers to billing staff, it passes intent, verification status, attempted actions, and error context so the user does not have to retell the story from scratch. This is where automation either builds trust or shatters it.
TriggerEscalation BehaviorPayload Sent
Repeated auth failureWarm transfer to billingSession ID, auth_fail, no PHI shown
API unavailableWarm transfer or callbackIntent, affected API, retry count
Dispute / refund / fraudImmediate human routingHigh-risk tag, user utterance summary
Payment failure x2Offer agent or alternate follow-upCharge code, amount, payment step reached
✓
Handoff Standard: The assistant should summarize the transfer aloud in one sentence. Example: "I'm connecting you with billing and letting them know you were trying to make a payment but the transaction didn't go through."
LLM Guardrails
  • No fabricated balances: The assistant must never generate a dollar amount or due date without API confirmation.
  • No clinical guidance: If the user shifts into medical advice, route away from billing and escalate appropriately.
  • RAG for policy, API for transactions: FAQs may use retrieval. Billing facts may not.
  • No account existence confirmation pre-auth: Friendly language must stay generic until identity is verified.
  • Commlog every session: Every interaction writes back to the system for continuity and auditability.
API Reference Map
  • OAuth Token
    POST /oauth/token
    Acquire bearer token for downstream billing and notification services.
  • Patient Lookup
    GET /patients?phone={ANI}
    Attempt pre-auth identity resolution using ANI or known channel identifier.
  • Identity Verification
    POST /auth/verify
    Validate DOB or alternate identifier before exposing any billing data.
  • Balance Retrieval
    GET /billing/balance?patientId=
    Return total due, due date, delinquency flags, and payment-plan eligibility.
  • Payment Tokenization
    POST /payments/tokenize
    Convert payment method to a secure token; never store raw details in transcript.
  • Charge Payment
    POST /payments/charge
    Execute the transaction using the secure token and approved amount.
  • Receipt Notification
    POST /notifications/sms
    Send confirmation when consent and channel settings allow it.
  • Communication Log
    POST /commlogs
    Persist outcomes, escalation reasons, and interaction metadata for staff continuity.
Open Items — Confirm Before Build
Critical — Blocks Payment Flow
Fee Policy: Confirm whether card, ACH, or channel convenience fees apply and whether they differ by payment amount or payer segment.
Critical — Blocks Go-Live
Payment Gateway Credentials: Tokenization and charge endpoints must be provisioned and tested in non-prod and prod environments.
Required — Policy Accuracy
Payment Plan Rules: Define minimum balance, eligibility flags, installment options, and when a human is required to finalize arrangements.
Required — Channel Logic
Receipt Preferences: Confirm whether SMS, email, or both can be offered by default and how opt-in / opt-out is stored.
Needed — Experience Tuning
Voice Tone Standard: Confirm whether HelixCare wants warmer concierge language or leaner utilitarian prompts for billing interactions.
Needed — Analytics
Revenue Attribution: Confirm whether completed AI payments should be attributed to containment, collections recovery, or overall digital self-service revenue.
Success Metrics
  • Containment: 65%+ of balance and simple payment requests resolved without human intervention.
  • Call Deflection: 30% reduction in billing-related live call volume.
  • Collections Lift: 15–20% increase in self-service payments over baseline.
  • Average Resolution Time: < 2 minutes for balance-only flows; < 4 minutes for successful payment flows.
