Denticon (PlanetDDS) — DFD Level 3

Appointment Management — Scheduling, Confirmation, Cancellation & Status Lifecycle

Source: Denticon API Reference Guide v1.0 · March 2026
Scope Definition
Included: All Denticon appointment management API endpoints documented in the Denticon API Reference Guide (March 2026): patient identity resolution, schedule availability lookup, appointment booking (POST Add Appointment), appointment status updates (PUT), cancellation, appointment notes updates, reminder flag updates, recall data retrieval, and supporting Setup API calls (Providers, Operatories, Appointment Types, Office Info). Both Legacy API (Auth Key) and New API (Bearer Token / Azure APIM) endpoint coverage is represented where documented.

Excluded: No middleware, event bus, or internal Denticon database schema is assumed. No UI or front-end components. Ledger/financial, insurance, and clinical flows are out of scope except where an appointment directly references a financial identifier (APID on ledger entries). Batch export and Webhook configuration are noted as supplementary but are not part of the real-time scheduling flow modeled here. No async processing or queueing infrastructure is inferred beyond what is documented. Rate limits are not published by Planet DDS — confirm with Planet DDS prior to implementation.

Authentication: Legacy endpoints use AuthKey header per PGID/OID; New API (2024+) uses Authorization: Bearer {token} via Azure APIM. Every request must include PGID and OID scope parameters.
Key Design Constraints (from Denticon API Reference Guide)
No atomic reschedule: Denticon has no reschedule endpoint. Rescheduling requires two steps — PUT APS=Cancelled on the existing APID, then POST a new appointment. Always secure the new slot BEFORE cancelling the old one.  |  Slot availability required before booking: Always call GET Schedule Availability before POST Add Appointment. The API will reject bookings into occupied or invalid slots.  |  TemporaryPatientId ≠ full PID: AddPatientInformation returns a TemporaryPatientId — the office must finalize the record in Denticon admin before it becomes a full active patient.  |  PGID + OID required on every call: Both identifiers are mandatory on virtually all endpoints.  |  Rate limits: Not published by Planet DDS — confirm with Planet DDS partner team.  |  Date format: MM/DD/YYYY for dates; MM/DD/YYYYTHH:MM:SS for datetime fields throughout the API.  |  New API access: Write access (scheduling, status updates) may require a higher-tier partner program enrollment via developer.planetdds.com. Confirm write permissions before building.

External Entities
E1
Voice Agent / IVA
Primary initiator of appointment management API calls. Performs patient lookup, slot search, booking, confirmation, cancellation, and status updates on behalf of the patient via the Denticon API. Operates within a PGID/OID scope established at session initialization.
E2
Patient / Caller
Human caller interacting with the Voice Agent. Provides identity credentials (name, DOB, phone) for lookup. Requests scheduling, confirmation, or cancellation actions. Does not interact with the Denticon API directly — all API calls are proxied through the Voice Agent.
E3
Dental Office / Front Desk Staff
Practice staff who manage provider schedules, configure appointment types, and finalize TemporaryPatientId records in Denticon admin. Receive appointment booking notifications from Denticon's configured reminder system. Set APS values for in-office transitions (Arrived, In Chair, Complete).
E4
Denticon Admin Portal
Practice management UI where Auth Keys are generated (Utilities > API Key Management), webhook endpoints are configured, and TemporaryPatientId records are reviewed and finalized. Not directly part of the real-time API flow but required for setup and patient record finalization.
E5
Denticon Webhook Receiver
Voice Agent's HTTPS endpoint registered with Denticon to receive push event notifications. Receives appointment.created, appointment.updated, appointment.status_changed, appointment.cancelled, and appointment.completed events. Must implement idempotent processing — Denticon retries failed deliveries.

5.0
Appointment Write & Update Operations
Book, confirm, cancel, update notes, update procedure codes, and manage reminder flags
READ + WRITE
POST /appointments/{PGID}/{OID}/add
Book Appointment (Add Appointment)
Create a new appointment in a specific available slot. Returns the APID of the newly created appointment. Triggers Denticon's reminder system if configured. APS defaults to 'Valid' on creation.
Required Fields
PGID OID PID SID (provider) OPID (operatory) APD (datetime MM/DD/YYYYTHH:MM:SS) APL (duration minutes) ATY (appointment type)
Optional
APN (notes) CDT codes ISNP (new patient flag) ISASAP (waitlist flag)
Response
APID (new appointment ID) APS (= Valid) Successful
⚠ Always call GET Schedule Availability immediately before this call. Never attempt to book without confirming slot is open — API will reject a booking into an occupied slot.
PUT /appointments/{PGID}/{OID}/{APID}/status
Update Appointment Status (Confirm / Cancel)
Update APS to any valid status value. Most common voice agent transitions: Valid → Confirmed (after verbal confirmation), Valid/Confirmed → Cancelled (per patient request). APS values: Valid, Confirmed, Arrived, In Chair, Complete, Cancelled, No Show, Missed, ASAP.
Required Fields
PGID OID APID APS (new status value)
Optional (for Cancellation)
cancellation reason
Response
Successful APID APS (updated)
⚠ To cancel: pass APS='Cancelled' + cancellation reason. Frees the slot immediately. Offer to reschedule BEFORE cancelling.
PUT /appointments/{PGID}/{OID}/{APID}/notes
Update Appointment Notes
Update or add notes to an appointment. APN field. Use to document reason for visit, patient-stated preferences, or voice agent interaction details captured during the call.
Required Fields
PGID OID APID APN (note text)
Response
Successful APID
PUT /appointments/{PGID}/{OID}/{APID}/procedures
Update Appointment Procedure Codes
Update procedure codes (CDT/ADA codes) on an existing appointment. Affects fee calculation. Primarily a clinical staff operation — voice agent reference only.
Required Fields
PGID OID APID CDT codes array
Response
Successful
ⓘ Clinical staff use — voice agent reads procedure codes from appointment but typically does not update them.
PUT /appointments/{PGID}/{OID}/{APID}/reminder-status
Update Appointment Reminder Flags
Update reminder-sent flags on an appointment. ISC (confirmed), ISM (messaged), ISP (portal notified), ISN (notified). Mark reminder sent after the IVA sends a confirmation message.
Required Fields
PGID OID APID reminder flag(s)
Reminder Flags
ISC (confirmed) ISM (messaged) ISP (portal notified) ISN (notified)
Response
Successful
6.0
Reschedule — Two-Step Operation
No atomic reschedule exists in Denticon — cancel existing APID then book new slot
READ + WRITE
GET /appointments/availability
Step 1: Find New Available Slot
Before cancelling the existing appointment, identify and confirm a new available slot. Offer 2-3 specific new times to the patient. Secure patient agreement on the new slot FIRST. If no new slot is available or acceptable, offer the ASAP waitlist instead of proceeding with cancellation.
Required Parameters
PGID OID SID date range ATY
⚠ Secure new slot agreement BEFORE cancelling. Slots are not held — proceed immediately to Steps 2 and 3 after patient agrees.
PUT /appointments/{PGID}/{OID}/{APID}/status
Step 2: Cancel Existing Appointment
Set the existing appointment APS = 'Cancelled'. Include cancellation reason. The slot is freed immediately after this call. Proceed immediately to Step 3 — do not delay between cancel and rebook.
Required Fields
PGID OID APID (existing) APS = 'Cancelled' cancellation reason
POST /appointments/{PGID}/{OID}/add
Step 3: Book New Appointment
Book the new appointment into the agreed slot immediately after cancellation. Returns new APID. Set APS = 'Confirmed' in a follow-up PUT call. Read back the new appointment details (date, time, provider, office address) to the patient for confirmation.
Required Fields
PGID OID PID SID OPID APD (new datetime) APL ATY
Response
APID (new) APS (= Valid)
⚠ Slot availability not guaranteed between cancel and rebook. If book fails, inform patient and re-search immediately.
7.0
Webhooks — Appointment Event Notifications
Push-based real-time event notifications for appointment changes — eliminates polling for production deployments
READ (Event-Driven)
POST Denticon → {your-webhook-endpoint}
Appointment Webhook Events (Inbound)
Denticon sends HTTP POST to your registered HTTPS endpoint when appointment events occur. Events: appointment.created, appointment.updated, appointment.status_changed, appointment.cancelled, appointment.completed. Payload contains event type, timestamp, PGID, OID, and APID. Fetch full appointment via GET by APID after receiving event.
Webhook Payload
event type timestamp PGID OID APID
Webhook Configuration (via Admin Portal)
endpoint URL (HTTPS) shared secret / signature event categories PGID/OID scope
⚠ Implement idempotent event handling — Denticon retries failed webhook deliveries. Use the Denticon event ID as deduplication key. Validate shared secret on every received event.
Denticon Data Stores (Domain Objects)
D1 Appointment Record
Accessed via: GET /appointments/{PGID}/{OID}, GET /appointments/{PGID}/{OID}/{PID}, GET /appointments/{PGID}/{OID}/{APID}, POST /add, PUT /status, PUT /notes, PUT /procedures, PUT /reminder-status
Core appointment record keyed by APID (Appointment ID). Unique within PGID/OID scope. Lifecycle controlled by APS field. Linked to a patient (PID), provider (SID), and operatory (OPID). Created by POST Add Appointment; mutated by PUT status/notes/procedures/reminder-status. Freed (slot available) when APS = 'Cancelled'.
APID PGID OID PID SID (provider) OPID APD (datetime) APL (duration) ATY (type) APS (status) APN (notes) CDT codes fee ISNP ISASAP ISC / ISM / ISP / ISN CBY / CBO MBY / MBO RS (Modified flag)
D2 Patient Record
Accessed via: POST /registration/check-existing-patient, GET /patients/v2/{PGID}/{OID}/{PID}, GET /patients/v2/recall
Patient identity and demographics record keyed by PID (Patient ID) within PGID/OID. RID (Responsible Party ID) is the financially responsible party — may differ from PID. Contains contact consent flags (ISAS, ISAE), active status (AC), patient type (PTY), medical alerts (MEDA), and inline insurance array. Primary key for all appointment and scheduling calls.
PID RID PGID OID FN / LN / MI BD SX HP / WP / CP EM AD / CT / ST / ZP AC PTY MEDA ISAS ISAE ISCD FV / LV Insurance[] BLC / CURRBLC OVR30 / OVR60 / OVR90 / OVR120 PN
D3 Schedule / Provider Availability
Accessed via: GET /appointments/availability, GET /setup/provider-blocked-times, GET /setup/appointment-slots
Provider schedule data defining when appointments can be booked. Scoped by PGID/OID/SID/OPID. Includes open slot start times, durations, operatory assignments, blocked times (out-of-office, holidays, lunch), and slot template priority settings. Read-only from API perspective — written by Denticon admin/practice staff via the admin portal.
SID (PDR) OPID available start times APL (duration) DH (blocked date) TY (block type) DS / DE (block start/end) LUS / LUE (lunch) EFD (effective date) STS (slot start) DU (slot duration) PRTY
D4 Practice Setup Configuration
Accessed via: GET /setup/office-info, GET /setup/providers, GET /setup/operatories, GET /setup/appointment-types, GET /setup/holidays, GET /setup/referral-sources
Static practice configuration data cached at session initialization. Contains office location details, provider list, operatory definitions, appointment type codes and durations (APNDUR), holiday dates, and referral sources. Rarely changes intra-day — refresh every 4-8 hours is sufficient.
PGID / OID OFN (office name) office address phone / fax TZ (timezone) WS (website) PDR / SID provider name OPID / OPN ATY codes APNDUR holiday dates referral source codes
D5 Recall Record
Accessed via: GET /patients/v2/{PGID}/{OID}/recall
Patient recall and continuing care records. Keyed by PID and procedure code. Contains recall interval (RCINT months), recall due date (RDT), and last scheduled date (SCHDT). Used to verify recall is actually due before offering a booking during outreach calls. ISCD field on patient record indicates continuing care dental flag.
PID PGID / OID procedure code RCINT (recall interval months) RDT (recall date) SCHDT (scheduled date)

Appointment Booking — Recommended API Sequence
The following sequence reflects the step-by-step booking flow documented in the Denticon API Reference Guide (March 2026), Section: "Appointment Booking — Step-by-Step Flow." All steps reference documented API endpoints only.
1
GET /setup/providers + operatories + appointment-types + office-info Session Initialization — Cache Setup Data
At session start, call all Setup API endpoints and cache responses. Providers (PDR/SID), operatories (OPID), appointment types (ATY + APNDUR), office info (OFN, address, TZ), and holidays. Refresh every 4-8 hours — do not call per-appointment. TZ field is required for correct datetime handling.
PGID OID PDR/SID (cached) OPID (cached) ATY + APNDUR (cached) TZ (cached)
2
POST /registration/check-existing-patient Identify Patient
Search for existing patient using FirstName, LastName, BirthDate. Returns IsExistingPatient flag and PID. Run on EVERY inbound call before any scheduling action. Prevents duplicate patient creation.
FirstName LastName BirthDate PGID OID phone
3
GET /patients/v2/{PGID}/{OID}/{PID} Get Full Patient Record
Pull complete patient record after PID confirmed. Verify AC=true (active patient). Check MEDA for medical alerts. Verify identity verbally (2 of: DOB, last 4 of phone, zip code) before accessing PHI.
PID PGID OID AC (verify = true) MEDA (check alerts) ISCD (recall flag)
4
GET /appointments/{PGID}/{OID}/{PID} Check Existing Appointments
Retrieve all upcoming appointments for the patient. Check for existing upcoming appointments to avoid double-booking. If a future appointment exists, confirm with patient whether they want to keep it, modify it, or book an additional appointment.
PGID OID PID date range filter
5
GET /patients/v2/{PGID}/{OID}/recall (if recall call) Verify Recall Status (Recall Calls Only)
For recall outreach calls, verify RCINT and RDT to confirm recall is actually due. Check SCHDT to ensure patient does not already have a scheduled recall appointment.
PGID OID PID RCINT RDT SCHDT
6
GET /setup/provider-blocked-times + GET /appointments/availability Find Available Slots
First check provider blocked times (out-of-office, holidays) to filter the search range. Then call GET Schedule Availability with PGID, OID, SID (provider), date range, and ATY (appointment type from cached setup). Returns open start times, OPID, and duration. Present 2-3 specific times to patient — keep list fresh.
PGID OID SID (from cached providers) date range ATY (from cached types) OPID
7
POST /appointments/{PGID}/{OID}/add Book Appointment
After patient confirms their selected slot, immediately POST the new appointment. Required: PID, SID, OPID, APD (datetime in MM/DD/YYYYTHH:MM:SS format), APL (duration from APNDUR), ATY (appointment type). Returns APID. APS defaults to 'Valid' on creation.
PGID OID PID SID OPID APD APL (= APNDUR from type) ATY APN (notes) ISNP
8
PUT /appointments/{PGID}/{OID}/{APID}/status Set Status to Confirmed
Immediately after receiving the new APID, PUT the appointment status to 'Confirmed'. This transitions APS from 'Valid' (default on booking) to 'Confirmed' reflecting verbal patient acceptance on the call.
PGID OID APID APS = 'Confirmed'
9
GET /appointments/{PGID}/{OID}/{APID} Read Back Appointment Details to Patient
Pull the full appointment record by APID and read back: date, time, provider name, office name, office address (from cached setup), and any pre-appointment instructions. Confirm patient has the correct information.
PGID OID APID APD (read back) SID → provider name OFN + address (cached)
10
PUT /appointments/{PGID}/{OID}/{APID}/reminder-status Update Reminder Flags
After reading back confirmation to the patient and sending any confirmation message, update the appointment reminder flags: ISC=true (confirmed), ISM=true (messaged) if confirmation SMS/email was sent. Completes the booking flow.

Appointment Cancellation / Reschedule — Recommended API Sequence
Cancellation flow documented in Denticon API Reference Guide (March 2026), Section: "Appointment Cancellation — Step-by-Step Flow." Note: Denticon has NO atomic reschedule endpoint — rescheduling requires the full cancel + rebook two-step sequence shown below.
1
POST /registration/check-existing-patient → GET /patients/v2/{PGID}/{OID}/{PID} Identify & Verify Patient
Run CheckExistingPatient to get PID. Pull full patient record. Verify identity verbally (2 of: DOB, last 4 of phone, zip). Confirm AC=true before proceeding.
FirstName LastName BirthDate PID (result)
2
GET /appointments/{PGID}/{OID}/{PID} Find Upcoming Appointment (APID)
Retrieve all upcoming appointments for the patient. Identify the APID of the appointment to cancel. Confirm appointment details with patient before proceeding (date, time, provider).
PGID OID PID APID (identify target) APD / SID / ATY
3
GET /appointments/availability Offer to Reschedule FIRST (Proactive Deflection)
Before accepting cancellation, search for available replacement slots and offer 2-3 specific times. Proactive deflection significantly reduces true cancellation rate. If patient agrees to reschedule, proceed with the two-step reschedule flow (Steps 4b → 5b) rather than cancellation only.
PGID OID SID date range ATY
4
PUT /appointments/{PGID}/{OID}/{APID}/status (ISASAP flag) Offer ASAP Waitlist (If No Immediate Reschedule)
If patient declines all offered rescheduling slots, offer to add them to the ASAP/waitlist: set ISASAP=true on a new provisional booking. This converts some cancellations into deferrals. If patient accepts ASAP list, book a future fallback date with ISASAP=true and proceed to soft-cancel or defer the existing appointment.
APID (new provisional) ISASAP = true APD (future fallback date)
5
PUT /appointments/{PGID}/{OID}/{APID}/status (APS=Cancelled) Cancel Appointment
Set APS = 'Cancelled' on the appointment. Include cancellation reason (required for reporting). Slot is freed immediately. If rescheduling: execute this step immediately followed by POST new appointment (Step 6) — do not delay between cancel and rebook.
PGID OID APID APS = 'Cancelled' cancellation reason
6
POST /appointments/{PGID}/{OID}/add (if rescheduling) Book New Appointment (Rescheduling Only)
If patient agreed to reschedule, immediately POST the new appointment into the agreed slot. Return new APID. PUT status = 'Confirmed'. Read back new appointment details to patient. If cancellation only (no rescheduling), confirm the cancellation verbally and provide the office phone number for future rebooking.
PGID OID PID SID OPID APD (new datetime) APL ATY
